## ⚡ 核心指令

### 1. 任务执行流程（必读）

```
用户给需求
  → 分析任务复杂度（简单/复杂/架构级）
  → 简单任务：直接做，完成后报告
  → 复杂任务：先问"这个方案可以吗？"（含备选）
  → 架构级任务：输出执行计划，用户批准后再行动
```

**简单任务**（5 分钟内可完成）：
> 直接执行，无需确认。例如：修复一个 bug、添加一个字段、写一个函数。

**复杂任务**（需要多步或跨文件修改）：
> 先列出方案要点，让用户确认方向正确再做。

**架构级任务**（涉及整体设计、多模块改动）：
> 先输出完整执行计划，征求同意。

### 2. 代码质量守则

```
✅ 做：写有意义的代码，有清晰输入输出
✅ 做：保持函数小而专注（一个函数 ≤ 50 行）
✅ 做：变量命名见名知意
✅ 做：添加必要的错误处理
✅ 做：完成后写简短 TODO 注释标注待优化点
✅ 做：使用 git checkpoint 保护工作（复杂修改前）

❌ 不做：不写无意义的注释（不要 "// 这是一个函数"）
❌ 不做：不过度工程化（不要为小任务写接口层）
❌ 不做：不加 AI 签名或 "Generated by" 水印
❌ 不做：不在 commit 中使用 emoji
❌ 不做：不擅自修改 .gitconfig 或用户凭证
```

### 3. Think 强度分级（来自 Claude 官方）

| 场景 | 指令 | 适用 |
|------|------|------|
| 常规补全 | `think` | 简单代码补全 |
| 函数重构 | `think hard` | 复杂逻辑修改 |
| 架构设计 / Bug 根因 | `ultrathink` | 多步推理，CoT 思考 |

> 遇到顽固 bug → 先加 `ultrathink` 再提交，强制多步推理链。

### 4. Git 安全操作

**复杂修改前必做 checkpoint：**
```bash
git add -A && git commit -m "Pre-modification checkpoint" || echo "No changes"
```

**危险操作（三思而后行）：**
- 删除文件：`trash` > `rm`（可恢复）
- 大规模重构：先 git checkpoint，验证后合并
- 批量删除：列出清单，征得同意后再执行

### 5. 代码审查（当被要求 /review 时）

使用多子代理并行分析：
- **安全子代理**：凭证暴露、输入校验、漏洞
- **性能子代理**：瓶颈、内存问题、优化机会
- **质量子代理**：复杂度、可维护性、最佳实践
- **架构子代理**：分层、依赖方向、可扩展性

输出格式：
```
🔴 严重 | 🟡 中等 | 🟢 建议
文件:行号 — 问题描述 → 修复方案
```

审查后询问："需要为严重问题创建 GitHub Issues 吗？"

## 🛠️ 自定义斜杠命令（可扩展）

| 命令 | 用途 |
|------|------|
| `/clean` | 清理调试产物，保留工作代码（带 git checkpoint） |
| `/scaffold [功能名]` | 根据项目模式生成完整功能模块 |
| `/explain` | 解释代码（类比 senior 工程师讲法） |
| `/test` | 运行测试并分析失败原因 |
| `/understand` | 分析整体项目架构和模式 |
| `/commit` | 智能 conventional commit |
| `/review` | 多维度代码审查 |
| `/predict` | 主动发现问题并预估修复时间 |
