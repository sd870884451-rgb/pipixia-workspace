## 📌 一、项目初始化阶段

### 1.1 全新项目启动

```
请帮我初始化一个 [语言/框架] 项目，要求：
- 项目名称：[名称]
- 技术栈：[版本号要具体，如 Python 3.11 + FastAPI + SQLAlchemy 2.0]
- 功能：简短描述核心功能
- 风格：约定俗成的项目结构，不要过度设计
```

**关键技巧：版本号要具体** → 明确版本号可规避 80% 的过时代码问题。

### 1.2 项目接手（已有代码库）

```
请分析当前项目：
1. 整体架构和模块划分
2. 使用的技术栈和依赖版本
3. 核心业务逻辑（用一句话概括）
4. 待改进的地方（按优先级排序）

先读 README.md 和主要入口文件，再用 Glob/Grep 了解全貌。
```

### 1.3 模板安装（claude-code-templates 模式）

```bash
# 自动检测项目类型，安装对应配置
npx claude-code-templates@latest
```

## 📌 二、日常开发提示词

### 2.1 添加新功能

**弱提示（容易过度工程化）：**
```
添加一个登录功能
```

**强提示（来自 CCPlugins 实战）：**
```
使用 [具体技术栈] 创建登录功能：
- 表单包含邮箱格式校验
- 调用 /api/auth 接口
- 处理 401 错误并提示"登录失效"
- 不使用任何 UI 框架，原生 HTML+CSS 即可
- 先在 [文件路径] 写接口，接口可用后再写前端
```

### 2.2 修复 Bug

**弱提示：**
```
修复这个 bug
```

**强提示（CCPlugins 风格 + ultrathink）：**
```
[ultrathink]
修复 [文件:行号] 的 bug：
- 预期行为：[描述]
- 实际行为：[描述]
- 已排除的原因：[列出你尝试过的分析]
- 先不要改代码，先分析根因，给出假设
- 假设通过后再修改，每次修改后运行 [测试命令] 验证
```

### 2.3 代码重构

**来自 CCPlugins /refactor：**
```
[think hard]
重构 [模块/函数]，目标：
- 函数行数控制在 50 行以内
- 消除重复逻辑，提炼公共函数
- 保持对外接口不变（不改变调用方式）
- 修改前先 git checkpoint
- 修改后运行测试验证
- 输出一个旧函数名→新函数名的映射表
```

### 2.4 批量修改

**跨文件全局上下文（"上帝视角"）：**
```
分析 [src/services] 下的所有文件：
1. 统一增加异常日志记录
2. 确保符合全局 Logger 规范（参考 [logger文件]）
3. 列出每个文件的改动清单，征求同意后再执行
```

## 📌 三、代码审查

### 3.1 完整审查（CCPlugins /review 模式）

```
使用多子代理并行分析项目：
- 安全子代理：凭证暴露、输入校验、漏洞扫描
- 性能子代理：瓶颈、内存问题、优化机会
- 质量子代理：复杂度、可维护性、最佳实践
- 架构子代理：分层、依赖方向、可扩展性

输出格式（按严重程度）：
🔴 严重 [文件名:行号] — 问题 → 修复方案
🟡 中等 [文件名:行号] — 问题 → 建议

审查后询问：需要为严重问题创建 GitHub Issues 吗？
```

### 3.2 安全专项扫描

```
[think hard]
扫描 [模块/文件] 的安全问题：
1. 凭证/密钥硬编码
2. SQL / 命令注入风险
3. 未校验的用户输入
4. 权限绕过漏洞
5. 敏感数据泄露（日志、错误信息）

输出：问题列表 + 严重程度 + 修复代码
```

## 📌 四、Git / 项目管理

### 4.1 智能 Commit

```
分析当前 git diff，生成符合 Conventional Commits 规范的 commit message：
- feat: 新功能
- fix: 修复
- refactor: 重构
- docs: 文档
- test: 测试
- chore: 杂项

只描述"做了什么改变"，不描述"为什么改变"（那是 PR description 的事）。
不要用 emoji。
```

### 4.2 清理项目（CCPlugins /cleanproject）

```
[think]
清理项目中的调试产物：
1. 先 git checkpoint 保护工作
2. 识别目标（*.log, *.tmp, __pycache__, .cache 等）
3. 保护关键目录（.git, .claude, node_modules, vendor）
4. 显示将删除的清单，征求同意后再执行
5. 验证项目完整性
```

### 4.3 TODO 追踪

```
扫描项目中所有 TODO/FIXME/NOTE 注释：
1. 按文件分组
2. 按优先级排序（高：影响功能 / 中：代码质量 / 低：优化机会）
3. 估算修复时间
4. 询问：需要为高优先级项创建 GitHub Issues 吗？
```

## 📌 五、提示词工程核心原则

来自 Claude 官方 Prompt Engineering 指南：

### 5.1 高质量提示词三维度

| 维度 | 错误示范 | 正确示范 |
|------|---------|---------|
| **上下文明确化** | "写一个登录功能" | "使用 React Hooks + TS 创建登录表单，包含邮箱格式校验，调用 /api/auth 并处理 401" |
| **技术栈具体化** | "用 Python 写后端" | "用 Python 3.11 + FastAPI + SQLAlchemy 2.0，异步模式" |
| **约束显性化** | "不要出问题" | "仅包含当前版本字段，按 OpenAPI 3.0 输出" |

### 5.2 约束正向化

```python
# ❌ 错误：负向约束，AI 推理歧义高
prompt = "生成API文档，不要包含过时字段，避免使用术语"

# ✅ 正确：正向约束，执行准确率高
prompt = "生成API文档，仅包含当前版本支持的字段，使用行业标准术语，OpenAPI 3.0 格式"
```

### 5.3 Think 强度矩阵

| 场景 | 指令 | Token 消耗 | 速度 |
|------|------|-----------|------|
| 常规补全 | `think` | 低 | 快 |
| 函数重构 | `think hard` | 中 | 中 |
| 架构设计 / Bug 根因分析 | `ultrathink` | 极高 | 慢 |

### 5.4 结构化输出

```
用 XML 标签包裹不同部分：
<分析>
[你的分析过程]
</分析>
<结论>
[你的结论]
</结论>
<代码>
[具体代码]
</代码>
```

## 📌 六、Multi-Agent 协作模式

来自 GitHub: Prorise-cool/Claude-Code-Multi-Agent

### 6.1 项目感知框架

```
项目配置阶段（.mcp.json）：
- 添加项目上下文感知
- 配置多代理协调
- 设定权限边界
```

### 6.2 分阶段执行（来自 andybhall 学术研究模式）

```
Phase 0: 项目初始化
→ 需求确认

Phase 1: 调研/文献综述
→ 批准后继续

Phase 2: 核心实现
→ 分步骤提交，用户批准每阶段

Phase 3: 测试验证
→ 全通过后再继续

Phase 4: 文档/交付
```
